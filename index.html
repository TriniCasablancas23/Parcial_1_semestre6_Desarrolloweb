1. Tecnologías de Web APIs

Arquitectura & Estándares

¿Qué es una API?

Las APIs (Application Programming Interfaces) son mecanismos que permiten a dos componentes de software comunicarse mediante protocolos definidos. Actúan como una capa intermedia que procesa transferencias de datos entre sistemas, permitiendo a las empresas abrir sus funcionalidades de manera segura y controlada.

🔄 REST (Representational State Transfer)

Estilo arquitectónico introducido por Roy Fielding en su tesis doctoral (2000). No es un protocolo sino un conjunto de principios orientados a la eficiencia, escalabilidad y simplicidad.

🖥️
Arquitectura Cliente-Servidor: Separa las preocupaciones de la interfaz de usuario de las de almacenamiento y lógica de negocio.
🔁
Sin Estado (Stateless): Cada petición debe ser completamente independiente; el servidor no guarda contexto entre solicitudes.
⚡
Capacidad de Caché: Mejora el rendimiento definiendo qué respuestas pueden reutilizarse para evitar peticiones redundantes.
🗂️
Sistema por Capas: Permite el uso de intermediarios como balanceadores de carga, proxies y gateways transparentes.
📨 SOAP

(Simple Object Access Protocol)

Protocolo de comunicación estrictamente basado en XML. A diferencia de REST, define reglas muy rígidas para el envío de mensajes. Es independiente del lenguaje y del protocolo de transporte (HTTP, SMTP o JMS).

2. Arquitectura de Microservicios

Frente a las aplicaciones monolíticas, los microservicios proponen dividir inteligentemente la aplicación en servicios pequeños, independientes y desplegables de forma separada.

¿Qué son?

Enfoque donde cada servicio gestiona una lógica de negocio única

La arquitectura de Microservicios es un enfoque de diseño donde una aplicación se construye como un conjunto de servicios pequeños, independientes y desplegables por separado. A diferencia de la arquitectura "monolítica", donde todo el código está en un solo bloque, aquí cada servicio se encarga de una función de negocio específica (por ejemplo: pagos, inventario, usuarios).

Esta independencia permite que cada servicio sea escrito en un lenguaje de programación diferente y que se escale de forma individual. Si el módulo de pagos recibe mucho tráfico, puedes replicar solo ese servicio sin copiar toda la aplicación.

Escalabilidad Resiliencia Políglota Agilidad
✅ Escalabilidad Granular

Si solo el módulo de "Procesamiento de Pagos" está saturado, escalas únicamente ese servicio. En un monolito tendrías que replicar toda la aplicación, gastando recursos innecesariamente.

✅ Resiliencia (Aislamiento de Errores)

Si un microservicio falla (recomendaciones), el resto sigue funcionando. El usuario quizás no vea recomendaciones, pero aún puede comprar y ver su carrito.

✅ Libertad Tecnológica

Python para IA, Node.js para chat en tiempo real y Go para alta velocidad, todo dentro de la misma plataforma.

✅ Despliegue Independiente

Un equipo actualiza "Notificaciones" sin riesgo de tirar "Inventario". Ciclos de vida completamente separados.

⚠️ Desafíos Críticos
❌ Complejidad Operativa

Gestionar docenas o cientos de pequeñas apps requiere orquestadores (Kubernetes), monitoreo avanzado y logs centralizados.

❌ Latencia de Red

En monolito las funciones se llaman en memoria (instantáneo). En microservicios, cada comunicación es una llamada HTTP/gRPC que añade milisegundos.

❌ Consistencia de Datos

Cada servicio tiene su propia base de datos. Mantener integridad requiere patrones complejos como Saga para transacciones distribuidas.

❌ Depuración Difícil

Rastrear un error que atraviesa cinco servicios requiere "trazabilidad distribuida" para entender qué pasó y en qué microservicio ocurrió.

3. Desarrollo de una Web API

Implica el proceso técnico de construir los puntos de entrada (endpoints) de un sistema. Comienza con el diseño del modelo de datos y la definición de rutas (URLs). Es crucial considerar la seguridad mediante tokens como JWT y documentar con Swagger/OpenAPI.

  • Definición de Objetivos: Es fundamental comprender los problemas que la API intentará resolver y quiénes serán sus usuarios finales (desarrolladores internos, socios, público).
  • Análisis de Requerimientos: Identificar tanto los requisitos funcionales (qué hace la API) como los no funcionales (rendimiento, seguridad, disponibilidad), incluyendo los datos que se manipularán.
  • Planificación del Proyecto: Asignación de recursos, estimación de costos y tiempos, y elección de la metodología de desarrollo (Agile, Scrum, Kanban).

Estándares y Convenciones: Seguir principios REST y usar JSON/XML facilita que otros desarrolladores entiendan y consuman la API.

Métodos HTTP:

GET — Obtener datos POST — Crear recursos PUT — Actualizar DELETE — Eliminar

Ejemplo: GET /posts lista artículos · POST /posts crea uno · DELETE /posts/{'{id}'} elimina por ID

  • Uso de Frameworks: Express.js para Node.js, Flask o FastAPI para Python, Spring Boot para Java. Aceleran el desarrollo con patrones probados.
  • Autenticación y Autorización: Implementar mecanismos como JWT, OAuth 2.0 para verificar identidad (autenticación) y permisos (autorización) y evitar accesos no autorizados.
  • Versionamiento: Permite introducir mejoras sin afectar a usuarios que dependen de versiones anteriores. Ejemplo: /api/v1/users vs /api/v2/users.

  • Pruebas (Testing): Pruebas unitarias (componentes individuales) y de integración para asegurar que el sistema maneje errores correctamente en todos los escenarios.
  • Documentación Exhaustiva: Esencial para que otros sepan cómo usar la API. Herramientas como Swagger/OpenAPI o Postman generan documentación automática con ejemplos y casos de error.
  • Despliegue y Monitoreo: La API se implementa en producción buscando escalabilidad. Una vez "en vivo", se requiere monitoreo constante para detectar problemas de eficiencia o seguridad.
  • Monetización: Si la API es para uso externo, puede publicarse en mercados especializados (API Marketplaces) para generar ingresos.

4. Manejo de Contenedores

📦 ¿Qué es un Contenedor?

Una unidad estándar de software que empaqueta el código de una aplicación y todas sus dependencias (bibliotecas, archivos de configuración, tiempo de ejecución) para que se ejecute de forma rápida y confiable de un entorno a otro.

🔒 Aislamiento

A diferencia de las VMs, los contenedores no incluyen un SO completo. Comparten el kernel del SO anfitrión, haciéndolos increíblemente ligeros (MB vs GB).

🚀 Portabilidad

Un contenedor creado en la laptop de un desarrollador funcionará exactamente igual en un servidor de pruebas o en infraestructura de nube.

🐳 Docker

La plataforma que popularizó los contenedores. Permite automatizar el despliegue de aplicaciones dentro de contenedores de software.

Dockerfile

Archivo de texto con instrucciones para construir una imagen. Ej: "Usa Python 3.9", "Copia mi código", "Ejecuta este comando".

Imagen

Paquete ejecutable e inmutable con todo lo necesario para correr la app. Es el "molde".

Contenedor

Instancia en ejecución de una imagen. Puedes tener 10 contenedores corriendo de la misma imagen.

💡 Ejemplo: Con Docker, creas una imagen que ya tiene Node.js instalado; cualquier servidor con Docker podrá correr tu API en segundos.

☸️ Kubernetes (K8s)

Si Docker crea y corre un contenedor, Kubernetes gestiona cientos al mismo tiempo en un clúster de servidores. Es el cerebro que decide dónde y cómo deben vivir los contenedores.

Pod

Unidad más pequeña en K8s. Puede contener uno o más contenedores que comparten almacenamiento y red.

Node (Nodo)

Máquina (física o virtual) donde se ejecutan los Pods.

Deployment

Define el "estado deseado". Ej: "siempre 3 copias de mi Web API". Si un contenedor falla, K8s lo reinicia automáticamente.

💡 Ejemplo: Tu API se vuelve viral. K8s detecta el tráfico y crea 5 nuevos Pods automáticamente para repartir la carga.

5. Cloud-Native Development

El enfoque Cloud-Native no significa tomar una app vieja y subirla a internet (Lift-and-Shift). Es diseñar específicamente para explotar elasticidad, automatización y resiliencia. Una app nativa de la nube está pensada para fallar y recuperarse sola.

☁️ Modelos de Implementación
Nube Pública

Recursos de AWS, Google Cloud o Azure compartidos entre clientes. Más económica y escalable, ideal para startups.

Nube Privada

Recursos exclusivos de una organización. Máximo control y seguridad. Preferida por bancos y gobiernos.

Nube Híbrida

Combinación de ambas. Datos confidenciales en privada + nube pública para picos de tráfico (Hot Sale).

🔧 Modelos de Servicio
IaaS — Infraestructura como Servicio

El proveedor da servidores y redes virtuales. Tú instalas el SO y todo lo demás. Ej: Amazon EC2

PaaS — Plataforma como Servicio

El proveedor da el entorno de ejecución. Solo subes tu código. Ej: Heroku, Google App Engine

SaaS — Software como Servicio

Software listo vía navegador. No programas nada. Ej: Gmail, Slack, Salesforce

⚖️ Ventajas y Desventajas

✅ Ventajas

  • Pago por uso: Solo pagas los minutos/segundos que el servidor está encendido. Sin compra de hardware.
  • Escalabilidad Infinita: De 100 a 1 millón de usuarios en minutos sin comprar hardware.
  • Actualizaciones Automáticas: El proveedor mantiene el hardware y software base al día.

❌ Desventajas

  • Vendor Lock-in: Construir con herramientas exclusivas de Amazon hace difícil migrar a Google Cloud.
  • Dependencia de Conexión: Si internet falla o el proveedor cae, tu negocio se detiene.
  • Seguridad y Privacidad: Datos en servidores de terceros requieren auditorías estrictas.

🔐 Protocolo de Seguridad JWT

Los JSON Web Tokens (JWT) son un estándar abierto (RFC 7519) para transmitir información de forma segura entre partes como un objeto JSON compacto y autoverificable.

1. Header (Encabezado)

Algoritmo utilizado (HS256, RS256) y tipo de token (JWT).

2. Payload (Carga útil)

Claims o declaraciones: datos del usuario, roles, expiración. No almacenes datos sensibles aquí, es solo Base64.

3. Signature (Firma)

Verifica que el token no fue alterado. Generada con la clave secreta del servidor.

⚠️ Tip de Seguridad: Nunca almacenes tokens en LocalStorage. Usa Cookies HttpOnly para prevenir ataques XSS. Los tokens deben tener tiempo de expiración corto.
